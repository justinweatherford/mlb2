# Daily Opp Weak Pregame Workflow

This document describes how to run and interpret the `core_home_opp_weak` lane each day.

---

## What This Lane Is

`core_home_opp_weak` is a frozen, observe-only pregame lane. It identifies home-team
moneyline bets where the brain has a positive signal and the opponent is historically weak
(opponent team strength bucket = lt_40).

**This lane is NOT live-traded.** It is paper-tracked only. Do not promote it to production
until it has been observed for at least one full season.

---

## Commands

### Normal daily run (called automatically from dev.bat)

```bat
python opp_weak_pregame_report.py --date 2025-06-15 --no-live-fetch
```

`--no-live-fetch` prevents HTTP requests to SBR. It uses the pre-cached SBR HTML if
available (written by earlier startup steps). If no cache exists for the date, all games
will be marked `blocked_missing_data` — this is correct behavior, not an error.

### Run with live SBR fetch (first run of the day, or if cache is missing)

```bat
python opp_weak_pregame_report.py --date 2025-06-15
```

### Manually regenerate today's report

```bat
python opp_weak_pregame_report.py
```

### Historical backfill (read-only; never re-run to change the frozen baseline)

```bat
python opp_weak_pregame_report.py --all-historical
```

## Paper Log Identity Key

The paper tracking log uses a priority-ordered dedup key to identify individual games:

| Priority | Key type | When used | Doubleheader-safe? |
|----------|----------|-----------|-------------------|
| 1 | `game_id` | Game has a stable MLB game identifier (e.g. `"COL@LAD_1"`) | **Yes** — Game 1 and Game 2 get different suffixes |
| 2 | `game_pk` | `game_id` absent but MLB Stats API integer is present | **Yes** — each game has a unique integer |
| 3 | `"unsafe"` + date + home + away | Neither `game_id` nor `game_pk` available | **No** — emits a `[WARN]` line; doubleheaders may collapse |

A `[WARN]` in the console output means a row has no `game_id` or `game_pk` and the
fallback key was used. This is unusual for modern data (post-2022) but may occur for
early historical rows. If you see a warning, check whether `game_id` exists on the card.

The identity key also includes `lane` so that future multi-lane tracking won't cross-pollinate.

## Status Meanings

| Status | Meaning |
|--------|---------|
| `paper_eligible` | Opening line ≤ 68.0% (-213 ML). Lane qualifies for paper tracking. |
| `observe_only` | Opening line 68.0%–70.5% (-213 to -239). In range but too close to max. Monitor only. |
| `blocked_by_price` | Opening line > 70.5% (-239 ML). Market price is beyond the conservative entry cap. |
| `blocked_missing_data` | No SBR opening line found for this game. Cannot assess entry price. |

**Status is determined solely by the SBR opening line (pre-decision).** Closing line,
Kalshi current price, and brain probability do NOT affect status.

---

## Max Entry Price

The maximum entry probability is **70.5% (≈ -239 ML)**.

This is derived from the historical lane hit rate (74.7%) with Bayesian shrinkage
(n=20 toward a 65% baseline) → 73.5% conservative estimate, minus a 3pp safety haircut.

**Do not enter above this price.** The lane has not been validated at higher prices, and
the safety haircut exists to protect against overfitting.

The `paper_eligible` threshold (68.0% / -213 ML) is set 2.5pp below max entry to give
a buffer against fill slippage.

---

## What Counts as Paper Eligible

A game is paper-eligible when:
1. It qualifies for the lane (home, side_score ≥ 0.40, not suppressed, opponent bucket = lt_40)
2. The SBR opening no-vig probability is available (not missing)
3. The opening probability is ≤ 68.0% (≤ -213 ML)

If (3) is met, the paper tracking log records the entry at the opening probability.

---

## CLV is Post-Hoc Only

The **Closing Line Value (CLV)** column shows how much the market moved from open to close.

> CLV = closing_prob − opening_prob

**CLV is labeled [POST-HOC ONLY] throughout the report and UI.** It is recorded for
analysis but is never used for eligibility, status, or any pre-decision logic.

Using CLV as a filter would be a lookahead violation — the closing line is only known after
the market closes, which is after the hypothetical decision time (pregame open).

---

## Contaminated Fields (Never Used for Decisions)

These fields appear in historical data but must NEVER be used for eligibility or status:

| Field | Why Contaminated |
|-------|-----------------|
| `team_no_vig_avg` | SBR **closing** line — only known after market closes |
| `sbr_home_no_vig_avg` | Alias for closing line |
| `market_edge_pp` | Derived from closing line |
| `actual_minus_market` | Derived from closing line |
| `implied_roi_pct` | Uses closing line as entry price |

These fields are written to the CLV / result columns only, clearly labeled POST-HOC.

---

## Interpreting the Kalshi Price Column

The `current_kalshi_mid` shown in the report and UI is read from the local `kalshi_mlb.db`
WebSocket snapshot database. It reflects the most recent bid/ask midpoint seen by the bot.

- If `—` or `n/a`: no snapshot available for this game in the DB.
- This price is **display only** — it does not affect status or eligibility.
- It is NOT live-fetched by `--no-live-fetch`; the SQLite DB is read regardless.
